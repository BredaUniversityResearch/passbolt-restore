# Passbolt Restore (Docker)

This project restores a Passbolt backup (originally from a native/source install) into a
Docker environment for testing or migration purposes, following the
[official Passbolt migration guide](https://www.passbolt.com/docs/hosting/migrate/server/ce/docker/).

SSL is terminated by an nginx container using your production Let's Encrypt certificates,
so the setup works transparently with the original domain and the Passbolt browser
extension — no changes to your production server required.

---

## Prerequisites

- Docker and Docker Compose installed
- PHP installed on the host (to run the restore script)
- A Passbolt backup consisting of:
  - A MySQL dump file (`.sql` or `.sql.gz`)
  - `passbolt.php` configuration file
  - `gpg/` folder containing `serverkey.asc` and `serverkey_private.asc`
  - `jwt/` folder containing `jwt.key` and `jwt.pem`
  - `ssl/` folder containing `fullchain.pem` and `privkey.pem` from Let's Encrypt

---

## Folder Structure

```
passbolt-restore/               ← your Docker project root
├── docker-compose.yml
├── .env                        ← generated by passbolt-restore.php, never committed
├── .gitignore
├── nginx.conf
├── README.md
├── passbolt-restore.php        ← run this to restore
└── restore/                    ← drop all backup files here
    ├── .gitkeep
    ├── <your_dump>.sql.gz      ← your MySQL dump (any filename, .sql or .sql.gz)
    ├── passbolt.php            ← your Passbolt config backup
    ├── gpg/
    │   ├── serverkey.asc          ← GPG public key
    │   └── serverkey_private.asc  ← GPG private key
    ├── jwt/
    │   ├── jwt.key                 ← JWT private key
    │   └── jwt.pem                 ← JWT public key
    └── ssl/                        ← production SSL certificates
        ├── fullchain.pem           ← Let's Encrypt fullchain certificate
        └── privkey.pem             ← Let's Encrypt private key
```

## Steps

### 1. Place your backup files

Copy your backup files into the `restore/` folder as shown in the structure above.

> **MariaDB automatically imports every `.sql` and `.sql.gz` file it finds in
> `/docker-entrypoint-initdb.d/` on first boot.** This means you can drop your dump file
> into `restore/` with any filename you like — no configuration needed. Both plain `.sql`
> and compressed `.sql.gz` files are supported.

### 2. Run the restore script

```bash
php passbolt-restore.php
```

The script performs these steps automatically:

1. Reads database credentials, `fullBaseUrl`, and domain from `restore/passbolt.php`
2. Writes `.env` for Docker Compose
3. Patches `restore/passbolt.php`:
   - Replaces the database host with `db` (the Docker service name)
   - Sets `ssl.force` to `false` (SSL is handled by nginx, not Passbolt)
4. Copies SSL certificates into the `cert_data` volume
5. Runs `docker compose up -d`
6. Waits for the Passbolt container to be running
7. Copies GPG and JWT keys into the container (`docker cp`)
8. Sets correct ownership (`www-data`) and permissions (`440`) on both key sets via `docker exec`
9. Fixes JWT directory permissions (`root:www-data`, `550`)

### 3. Monitor and wait for the database import

> **Note on database import time:** MariaDB automatically imports your dump on first boot,
> but depending on the size of your dump file this can take several minutes. During this
> time MariaDB does not accept connections and Passbolt will not be fully operational yet.
> You can monitor progress with `docker compose logs -f db` and wait for the line
> `MySQL init process done. Ready for start up.` before proceeding.

### 4. Point your local hosts file to the Docker host

Since you are testing with the original domain, redirect it to your Docker host on your
local machine only — production is unaffected.

Add the following line to your hosts file, replacing the IP with your Docker host's IP:

```
<docker-host-ip>    passbolt.yourdomain.com
```

- **Windows:** `C:\Windows\System32\drivers\etc\hosts`
- **macOS / Linux:** `/etc/hosts`

> Remember to remove or comment out this line when you are done testing.

### 5. Accept the self-signed certificate

Navigate to your Passbolt URL and log in.

### 6. Verify the restore

Run the built-in Passbolt healthcheck:

```bash
docker exec --user www-data <container> bin/cake passbolt healthcheck
```

All checks should pass. Your restored Passbolt instance is ready to test.

---

## Re-running the restore

To do a clean restore from scratch, tear everything down including volumes first:

```bash
docker compose down -v
php passbolt-restore.php
```

> **Warning:** `down -v` permanently deletes all volumes including the database. Only use
> this when you intend to re-import from the dump.

---

## Notes

- The `restore/` folder and `.env` are excluded from Git (see `.gitignore`) to prevent
  accidental commits of sensitive backup data.
- `APP_FULL_BASE_URL` is set as an environment variable even though `passbolt.php` also
  contains `fullBaseUrl`, because Docker environment variables take priority over
  `passbolt.php` and the Docker image defaults to `https://passbolt.local` if it is not set.
